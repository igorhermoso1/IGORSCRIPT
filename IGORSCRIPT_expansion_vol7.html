🔄 Event Loop & Concurrencia
JavaScript tiene un solo hilo, pero parece hacer mil cosas a la vez. El secreto es el event loop. Entenderlo es LA pregunta de entrevista más repetida del mundo.

Las 4 piezas del motor:

  • Call Stack: donde se ejecuta el código síncrono, una función encima de otra.
  • Web APIs: el navegador gestiona ahí los setTimeout, fetch, eventos... fuera del hilo de JS.
  • Cola de microtareas: los .then() de las promesas y queueMicrotask(). Tiene PRIORIDAD MÁXIMA.
  • Cola de macrotareas: setTimeout, setInterval, eventos de UI. Se atiende DESPUÉS de vaciar las microtareas.
console.log('1'); // síncrono
setTimeout(() => console.log('2'), 0); // macrotarea
Promise.resolve().then(() => console.log('3')); // microtarea
console.log('4');
// Salida: 1, 4, 3, 2 — ¡aunque el setTimeout tenga 0ms!

Regla de Igor para entrevistas: primero TODO el código síncrono, luego TODAS las microtareas (promesas), y solo entonces UNA macrotarea (setTimeout). Y vuelta a empezar.

🧬 Prototipos & Herencia Prototípica
Las clases ES6 son azúcar sintáctico. Por debajo, JS solo conoce una cosa: objetos que apuntan a otros objetos. Esa cadena es la herencia real.

La cadena de prototipos: cuando pides una propiedad que el objeto no tiene, JS la busca en su prototipo, y en el prototipo del prototipo... hasta llegar a null.

const animal = { respira() { return 'inhala...'; } };
const perro = Object.create(animal); // perro hereda de animal
perro.ladra = () => 'guau';

perro.ladra(); // 'guau' → propia
perro.respira(); // 'inhala...' → heredada por la cadena

Object.getPrototypeOf(perro) === animal; // true
perro instanceof Object; // true: Object.prototype está al final de la cadena

Con funciones constructoras y class: los métodos de una class viven en NombreClase.prototype, no en cada instancia. Mil instancias comparten UN solo método en memoria.

Nunca uses __proto__ en código real (está obsoleto). Usa Object.getPrototypeOf() y Object.create().

🎯 this, call, apply, bind
this no depende de DÓNDE se escribe la función, sino de CÓMO se llama. Cuatro reglas lo deciden todo.

Las 4 reglas de this (de menor a mayor prioridad):

  • 1. Llamada simple: f() → this es undefined (strict mode) o window.
  • 2. Llamada de método: obj.f() → this es obj (lo que hay antes del punto).
  • 3. Explícito: f.call(x), f.apply(x), f.bind(x) → this es x.
  • 4. Constructor: new F() → this es el objeto nuevo.

Excepción total — arrow functions: NO tienen this propio. Heredan el this del lugar donde fueron escritas. Por eso son perfectas para callbacks.

const mago = {
  nombre: 'Igor',
  saluda() { return `Soy ${this.nombre}`; }
};
mago.saluda(); // 'Soy Igor' — regla 2

const suelta = mago.saluda;
suelta(); // 'Soy undefined' — regla 1: ¡perdió el contexto!

const atada = mago.saluda.bind(mago);
atada(); // 'Soy Igor' — regla 3: bind lo fija para siempre

⚖️ Coerción de Tipos
JavaScript convierte tipos automáticamente cuando no le cuadran. Entender esas reglas evita el 80% de los bugs raros.

== convierte, === no: el doble igual transforma los operandos hasta que coincidan los tipos. El triple igual compara tipo Y valor sin tocar nada.

5 == '5'  // true (convierte el string a número)
5 === '5' // false (tipos distintos, fin)
null == undefined // true (caso especial del lenguaje)
null === undefined // false
NaN === NaN // false ¡NaN no es igual ni a sí mismo!
Object.is(NaN, NaN) // true — la comparación más estricta

Los 8 valores falsy: false, 0, -0, 0n, "", null, undefined, NaN. TODO lo demás es truthy, incluidos "0", [] y {}.

Regla de Igor: usa SIEMPRE ===. La única excepción aceptada en la industria es x == null para comprobar null y undefined a la vez.

🧠 Memoria, Garbage Collector & Weak*
JS libera memoria solo... salvo cuando tú se lo impides sin darte cuenta. Aquí aprendes a no crear memory leaks y a usar las colecciones débiles.

Garbage Collector (GC): un valor se libera cuando nada lo puede alcanzar. Si un objeto sigue referenciado (en una variable global, un Map, un listener...), NUNCA se libera.

const cache = new Map();
let usuario = { nombre: 'Igor' };
cache.set(usuario, 'datos');
usuario = null; // ¡el objeto sigue vivo dentro del Map! → leak

const cacheDebil = new WeakMap();
// con WeakMap, si el objeto-clave muere fuera, el GC lo limpia también dentro

WeakMap/WeakSet: solo aceptan objetos como clave, no son iterables y no tienen .size — precisamente porque sus entradas pueden desaparecer en cualquier momento.

Fuentes típicas de leaks: listeners no eliminados, intervalos sin clearInterval, caches con Map, closures que retienen objetos grandes.

🔢 Números Precisos & BigInt
0.1 + 0.2 no es 0.3 en JavaScript. No es un bug: es matemática binaria. Aquí aprendes a manejarla y a usar enteros gigantes.

IEEE-754: todos los números de JS son flotantes de 64 bits. Igual que 1/3 no se puede escribir exacto en decimal (0.3333...), 0.1 no se puede escribir exacto en binario.

0.1 + 0.2 // 0.30000000000000004

// Comparar flotantes: usa un margen de error
Math.abs((0.1 + 0.2) - 0.3) < Number.EPSILON // true ✓

// Enteros seguros solo hasta:
Number.MAX_SAFE_INTEGER // 9007199254740991 (2^53 - 1)

// Más allá: BigInt (sufijo n)
const gigante = 9007199254740993n;
gigante + 1n // exacto. Pero no se puede mezclar BigInt con Number

Para dinero, nunca uses flotantes: trabaja en céntimos con enteros (1999 en vez de 19.99) o usa BigInt. Los bancos no perdonan el 0.000000000004.

🌀 Recursión
Una función que se llama a sí misma. Imprescindible para árboles, JSON anidado, algoritmos de búsqueda y... entrevistas técnicas.

Anatomía de toda recursión:

  • Caso base: la condición que DETIENE las llamadas. Sin él → stack overflow.
  • Caso recursivo: la función se llama a sí misma con un problema MÁS PEQUEÑO.
function factorial(n) {
  if (n <= 1) return 1;  // caso base
  return n * factorial(n - 1); // caso recursivo
}
factorial(5); // 5*4*3*2*1 = 120

Memoización: guardar resultados ya calculados en un objeto/Map para no repetir trabajo. Convierte fibonacci de exponencial a lineal.

🫧 Eventos Avanzados
Ya sabes usar addEventListener. Ahora aprende cómo VIAJAN los eventos por el DOM y el patrón que usan todas las apps profesionales: delegation.

Un clic viaja en 3 fases:

  • Capturing: baja desde window hasta el elemento (poco usado).
  • Target: llega al elemento clicado.
  • Bubbling: sube de hijo a padre hasta window. Aquí escuchan casi todos los listeners.
// Event delegation: UN listener en el padre gestiona N hijos
lista.addEventListener('click', e => {
  const item = e.target.closest('li');
  if (item) console.log('clicaste', item.dataset.id);
});

// CustomEvent: tus propios eventos con datos
const ev = new CustomEvent('compra', { detail: { precio: 19.99 }, bubbles: true });
elemento.dispatchEvent(ev);

e.stopPropagation() frena el viaje del evento. e.preventDefault() cancela la acción nativa (enviar formulario, seguir enlace). Son cosas distintas.

⏱ AbortController, Timeouts & Retry
Un fetch que nunca responde puede colgar tu app. Aquí aprendes a cancelar peticiones, ponerles límite de tiempo y reintentarlas con cabeza.

AbortController: un mando a distancia para cancelar operaciones asíncronas.

const ctrl = new AbortController();
fetch(url, { signal: ctrl.signal });
ctrl.abort(); // el fetch lanza AbortError

// Timeout moderno en una línea:
fetch(url, { signal: AbortSignal.timeout(5000) });

// Timeout clásico con Promise.race:
const timeout = ms => new Promise((_, rej) => setTimeout(() => rej(new Error('Timeout')), ms));
await Promise.race([fetch(url), timeout(5000)]);

Retry con backoff exponencial: si falla, espera 1s y reintenta; si vuelve a fallar, 2s; luego 4s... Así no machacas un servidor caído.

🧪 structuredClone & JSON Profundo
Copiar objetos parece fácil hasta que un cambio en la copia rompe el original. Deep copy de verdad y los superpoderes ocultos de JSON.

El problema: const b = a y el spread {...a} NO copian en profundidad: los objetos anidados se comparten.

const a = { config: { tema: 'oscuro' } };
const b = { ...a }; // copia superficial
b.config.tema = 'claro';
a.config.tema; // 'claro' ¡también cambió el original!

// Deep copy moderna:
const c = structuredClone(a); // soporta Date, Map, Set, ciclos...

// Truco clásico (con limitaciones):
const d = JSON.parse(JSON.stringify(a)); // pierde funciones, undefined, Date→string

JSON avanzado: JSON.stringify(obj, replacer, 2) filtra/transforma al serializar; JSON.parse(str, reviver) transforma al leer; un método toJSON() en tu objeto decide cómo se serializa.

👁 MutationObserver & ResizeObserver
En vez de preguntar cada segundo "¿ha cambiado algo?", el navegador te avisa. Los observers son la forma eficiente de reaccionar a cambios.

MutationObserver: avisa cuando cambia el DOM (nodos añadidos/eliminados, atributos, texto).

const obs = new MutationObserver(mutaciones => {
  mutaciones.forEach(m => console.log('cambio:', m.type));
});
obs.observe(elemento, { childList: true, attributes: true, subtree: true });
obs.disconnect(); // siempre desconecta al terminar

ResizeObserver: avisa cuando un ELEMENTO cambia de tamaño (no solo la ventana como el viejo evento resize). IntersectionObserver (visto en los proyectos guiados) completa el trío: avisa cuando algo entra en pantalla.

// Si NECESITAS HTML: sanitiza (DOMPurify) o escapa los caracteres < > &

♿ Accesibilidad con JavaScript
Tu JS puede hacer una web usable para todos... o romperla para quien usa teclado o lector de pantalla. Esto distingue a un profesional.

Tres responsabilidades del JS accesible:

  • Focus management: al abrir un modal, mueve el foco dentro con elemento.focus(); al cerrarlo, devuélvelo a donde estaba. Atrapa el Tab dentro del modal.
  • ARIA dinámico: si tu JS cambia el estado, actualiza el atributo: btn.setAttribute('aria-expanded', 'true'). Las regiones aria-live anuncian cambios al lector de pantalla.
  • Teclado: todo lo clicable debe funcionar con Enter/Espacio. Si usas un div como botón (mal), necesitas tabindex="0" + listener de teclado. Mejor: usa <button>.

La primera regla de ARIA: no uses ARIA si un elemento HTML nativo ya lo hace. Un <button> real te regala foco, teclado y semántica gratis.

⚡ Rendimiento
Una app lenta pierde usuarios. Aquí están las técnicas que separan una web que vuela de una que se arrastra: debounce, reflow, rAF, virtual lists y Web Vitals.

Debounce vs Throttle: dos formas de domar eventos que se disparan decenas de veces por segundo (scroll, input, resize).

  • Debounce: espera a que el usuario PARE. Solo ejecuta tras X ms de silencio. Ideal para buscadores.
  • Throttle: ejecuta como máximo una vez cada X ms, aunque el evento siga. Ideal para scroll.
function debounce(fn, ms) {
  let timer;
  return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), ms);
  };
}

Reflow (layout thrashing): leer una medida del DOM (offsetHeight) justo después de escribir estilos obliga al navegador a recalcular TODO el layout. En un bucle, eso congela la página. Regla: agrupa todas las lecturas y luego todas las escrituras.

requestAnimationFrame: para animar, nunca setInterval. rAF sincroniza con el repintado (60fps), se pausa en pestañas ocultas y no produce tirones.

Web Vitals: las 3 métricas que mide Lighthouse: LCP (cuánto tarda en verse lo principal), CLS (cuánto salta el layout), INP (cuánto tarda en responder a una interacción).

Antes de optimizar, mide: pestaña Performance de DevTools y Lighthouse (F12 → Lighthouse → Analyze). Optimizar sin medir es adivinar.
